## Discovery: openpyxl never writes shared strings

**openpyxl 3.1.5 always writes string cells as `inlineStr` and never emits
`xl/sharedStrings.xml`** (see `openpyxl/cell/_writer.py`, lines 21‚Äì22 and 70‚Äì79).

Two consequences, both useful for T4 (shared-string parsing):

1. **`values-basic.xlsx` has no `sharedStrings.xml` at all** ‚Üí it doubles as the
   *"missing shared string table"* case (a valid workbook whose strings are all
   inline). T4 must handle this without error.
2. **`strings-shared.xlsx` is hand-crafted** minimal OOXML (not real producer
   output) so that a genuine shared string table exists to test ‚Üí the *"present
   shared string table"* case, with de-duplication and index resolution.

Together these two fixtures fully cover T4's read paths (present, missing, and
`inlineStr`). The hand-crafted file is structurally validated by round-tripping
through openpyxl; **OWNER will additionally open it in LibreOffice / Google
Sheets to confirm real-app compatibility.**

## Detailed golden values

### `strings-shared.xlsx`

Shared string table (`xl/sharedStrings.xml`, in order):

| Index | Value |
|---|---|
| 0 | `fruit` |
| 1 | `apple` |
| 2 | `banana` |
| 3 | `cherry` |

Cell ‚Üí index ‚Üí resolved value (column A):

| Cell | Index | Value |
|---|---|---|
| `A1` | 0 | fruit |
| `A2` | 1 | apple |
| `A3` | 2 | banana |
| `A4` | 1 | apple |
| `A5` | 3 | cherry |
| `A6` | 2 | banana |
| `A7` | 1 | apple |

`A2`, `A4`, `A7` all resolve to `apple` via the same index `1`.

### `sparse-gaps.xlsx`

0-based `(row, col)` coordinates the parser must produce (and nothing else):

| Cell | (row, col) | Value |
|---|---|---|
| `A1` | (0, 0) | "start" (text) |
| `C5` | (4, 2) | 42 (number) |
| `AA100` | (99, 26) | "end" (text) |

Column `AA` = index 26; row 100 = index 99.

### `dates.xlsx`

| Cell | Value | numFmtId | Kind | Excel serial |
|---|---|---|---|---|
| `A1` | 2024-01-15 | 14 (builtin) | date | 45306 |
| `A2` | 13:30:00 | 21 (builtin) | time | 0.5625 |
| `A3` | 2024-01-15 13:30:00 | 22 (builtin) | datetime | 45306.5625 |
| `A4` | 2024-12-31 | 164 (custom `yyyy-mm-dd`) | date | 45657 |

Note: openpyxl also *defines* an unused custom numFmt `165` (`yyyy-mm-dd
h:mm:ss`) that no cell references ‚Äî a harmless real-world artifact. Serials use
the 1900 epoch (`date1904=false`).

## Performance fixture

`big-50k-rows.xlsx` ‚Äî a producer-agnostic scale fixture for the M2 performance
target (~50,000 data rows √ó 7 columns; mostly numeric, with one repeated text
column and a boolean). It has **no golden values** ‚Äî it exercises scroll/render
throughput, not producer variance ‚Äî so it is generated separately and is not
part of the default 8-fixture run.

## Regenerating

```bash
pip install openpyxl        # tested with 3.1.5
python3 tools/gen_fixtures.py          # the 8 small golden fixtures
python3 tools/gen_fixtures.py big       # big-50k-rows.xlsx (large, perf only)
```

Output is (near-)deterministic: document timestamps and hand-crafted ZIP entry
times are pinned, so regeneration does not churn git. Exact ZIP bytes may still
vary across openpyxl versions, but the parsed golden values above will not.
